1. Overview

This document defines the technical design for a display/browsing-only product catalogue for Mangalist, at the business's actual current scale: two brands at launch (Char-Broil, Napoleon), roughly a couple of dozen SKUs, a two-person operations team (the twins), no orders yet, and no meaningful site traffic yet. The design deliberately does not build for hypothetical future scale — every place this document adds structure ahead of strictly proven need is called out explicitly, matching the DBA's own proportionality discipline in the schema design below.

The catalogue builds on the existing self-hosted Supabase stack (Postgres + PostgREST + RLS + Studio, run via Docker — locally today, with AWS EC2 as the already-agreed future deploy target) and the existing plain HTML/CSS/JS frontend (no framework, no build step, Hebrew/RTL). Architecture for this specific catalogue was explicitly reopened for reconsideration by the parent; this document evaluates alternatives and recommends, in almost every dimension, staying close to the existing pattern — with one genuine decision point (SEO / link-preview rendering for product pages) presented as an explicit either/or for the CIO/parent, not decided here.

2. Proposed Architecture

2.1 Frontend rendering & serving approach

Recommendation: keep the existing pattern — plain HTML/CSS/JS, client-side fetch from PostgREST, no framework, no build step.

Concretely: two new plain HTML pages, styled with the existing styles.css theme, sit alongside index.html:

  • catalog.html — browse/filter/search listing page. State (selected category, brand, price range, stock filter, search text, sort order) lives in the URL query string (e.g. catalog.html?category=gas-grills&brand=char-broil&stock=in_stock&q=...), so results are linkable/shareable and the page needs no server-side routing.
  • product.html — single product detail page, driven by an id in the query string (e.g. product.html?id=<uuid> — see §2.2 on why id/sku rather than a Hebrew slug for now).

Both pages use vanilla JS plus the @supabase/supabase-js CDN client already wired into index.html for the leads form, calling PostgREST directly with the public anon key. This is safe because RLS already restricts anon reads to published = true rows (same pattern as the existing products table policy) — no new trust boundary is introduced.

Rendering options considered
OptionDescriptionVerdict
A. Client-side fetch (recommended) Static HTML shell + vanilla JS fetches from PostgREST at page load, renders into the DOM. Zero new tooling, identical deploy story to today (static files + self-hosted Supabase). Recommended default. Proportionate to current traffic (near zero) and team size.
B. Build-time static pre-rendering A small Node.js generator script (not a framework — no bundler, no client hydration library) queries Postgres/PostgREST at build time and emits one static HTML file per published product, with real <title>/meta description/Open Graph tags baked in, run manually or on a schedule whenever the catalogue changes. Not recommended today — documented for later. See flag below. Real benefit: correct Google indexing and rich WhatsApp/Facebook link previews for individual product pages (client-side-only rendering means crawlers that don't execute JS — notably link-preview bots — see an empty page). Real cost: introduces a build step where none exists today, and a "someone must remember to regenerate" operational step.
C. Frontend framework (Next.js / Astro / Gatsby, SSR or SSG) Full framework with routing, data-fetching conventions, build pipeline. Rejected as over-engineering at this scale — a couple dozen SKUs, two brands, no orders, minimal traffic. Would add a real ongoing maintenance/learning-curve cost with no proportionate benefit over Option A or B.
RECOMMENDATION REQUIRING SIGN-OFF This design recommends Option A (no build step, matching the existing "no framework, no build step" principle) as the default for now, since there is no organic-search or social-sharing traffic yet to justify the extra moving part. Option B (build-time pre-rendering) is flagged as a genuine future decision point, not a rejected idea — once Marketing has an active plan to drive traffic to specific product links (SEO, WhatsApp/Facebook sharing), the CIO/parent should revisit whether the SEO/link-preview benefit now justifies the small added build step. No cost is attached to either option (no paid service), but Option B is a deviation from the current "zero build step" architecture and needs explicit sign-off before Dev implements it, whenever it is revisited.

2.2 URL / routing scheme (partially resolves DBA's Hebrew-slug open question)

The DBA flagged that "slug" is an English SEO convention that doesn't map cleanly onto Hebrew product names (open question 1, §4). This design proposes a split resolution, since brands/categories and products have very different slug-ability:

  • Brands and categories — small, curated, controlled lists (2 brands, a handful of categories). Recommend the twins/CIO assign real transliterated/English slugs (e.g. char-broil, napoleon, gas-grills, accessories) — low effort, done once, gives clean readable category/brand URLs today.
  • Products and variants — dozens of Hebrew product names with no obvious transliteration convention yet. Recommend routing by id (or sku for variants) for now (product.html?id=...), leaving the products.slug column null/unused until Marketing settles a Hebrew product-URL convention. This defers a real decision to the people who own site copy/positioning, rather than the Solution Architect guessing at it.

2.3 Search & filtering — how it's served to the browser

Per the DBA's recommendation, search and faceted filtering are both served directly by Postgres — no separate search service. The catalogue page's filters map directly onto PostgREST query parameters against the products/product_variants views: 

GET /rest/v1/products?select=id,name,slug,specs,brand:brands(name,slug),category:categories(name,slug),
    product_variants(id,sku,variant_name,price_ils,stock_status)
    &category_id=eq.<uuid>
    &brand_id=eq.<uuid>
    &product_variants.stock_status=eq.in_stock
    &fts_column=fts(simple).מנגל
    &published=eq.true
    &order=name.asc

Free-text search uses the DBA's generated tsvector column ('simple' config — Hebrew does not benefit from English stemming) via PostgREST's fts filter, exactly as recommended by the DBA. Facet filters (category, brand, price range, stock status) are plain equality/range filters using the existing indexes (category_id, brand_id, price_ils, stock_status). Facet counts (e.g. "Char-Broil (5)" badges next to filter checkboxes) are computed client-side from the already-fetched result set at this catalogue size — no materialized view or extra aggregation query is needed on day one, per the DBA's note that a materialized view is a cheap future addition only if heavy joined-aggregation queries become necessary later.

2.4 Product variants & stock/availability display

  • A product with exactly one variant (the common case for single-SKU accessories) shows no variant picker — price/stock come straight from that one product_variants row.
  • A product with >1 variant (e.g. a grill available in two sizes, or an accessory in multiple colors) shows a simple picker (dropdown or pill buttons) built from variant_name; selecting a variant updates the displayed price, stock badge, dimensions, and swaps in variant-specific images where present (product_images.variant_id), falling back to the product-level gallery otherwise.
  • Stock status badge, mapped from the DBA's stock_status enum to Hebrew UI labels:

Price display: if price_ils is null (expected at launch — no real ILS pricing exists yet per the DBA schema), show "מחיר לפי פנייה" (price on request) linking into the existing contact form, consistent with the site's current contact-driven approach rather than inventing a placeholder price.

2.5 Images — handling & serving

Recommendation: use the Storage service already included in the self-hosted Supabase stack (Postgres + PostgREST + Studio + Auth + Storage — confirmed already part of what supabase start brings up per supabase-local/README.md). A single public-read product-images bucket, with upload restricted to the postgres/service role (same bypass-RLS pattern already used for table writes via Studio). Twins/CIO upload product photos directly through Studio's Storage UI (drag-and-drop); the resulting public URL is stored in product_images.url.

Image storage options considered

The only cost implication is indirect: stored images add to the eventual AWS EC2 disk usage, which is already accounted for in the CIO's existing 60–150 NIS/month estimate for the single-instance deploy. Flagging for completeness, not because it's a new decision.

2.6 Deployment topology

Unchanged from the CIO's existing Phase 1 plan in ceo/strategy.md: the same self-hosted Supabase docker-compose stack (now with more tables and the Storage bucket in use) runs locally via Docker Desktop today, and is deployed as the same compose file to a single AWS EC2 instance later. The catalogue frontend pages (catalog.html, product.html) are additional static files served the same way index.html is today — no new hosting component, no new topology decision introduced by this catalogue design.

3.2 Relationships

  • categories.parent_id — self-referencing tree (2 levels in practice today)
  • products.brand_id / products.category_id — many-to-one
  • product_variants.product_id — one-to-many (every product has ≥1 variant; a trivial "Default" variant covers single-SKU products)
  • product_images.product_id — one-to-many
  • product_images.variant_id — optional many-to-one
  • Delete behavior: brand_id/category_id FKs are on delete restrict; product_id FKs on variants/images are on delete cascade

3.3 Indexing

  • products(category_id), products(brand_id), categories(parent_id)
  • GIN index on products(specs) using jsonb_path_ops
  • btree on product_variants(price_ils); GIN/btree on product_variants(dimensions) if needed
  • Generated tsvector column on products (name + description, 'simple' config — Hebrew doesn't benefit from English stemming), with GIN index for search
  • Index on product_variants(stock_status)
  • Indexes on published columns across brands/categories/products/product_variants (belt-and-braces at this row count)
  • Unique indexes: brands.slug, categories.slug, products.slug, product_variants.sku, plus the partial unique on product_images primary-image

3.4 Row-Level Security

Same pattern as the existing products table: RLS enabled on every table; anon/authenticated get SELECT-only on published rows via policy; no public insert/update/delete (catalog managed via Studio using the postgres role, which bypasses RLS); an explicit grant select is required per table in addition to the RLS policy (a known PostgREST gotcha).

  • brands / categories / products: using (published = true)
  • product_variants: using (published = true and exists (select 1 from products p where p.id = product_variants.product_id and p.published = true)) — a variant can be independently hidden but is never shown if the parent product is hidden
  • product_images: no own published column; using (exists (...product published...) and (variant_id is null or exists (...variant published...))) — visibility is fully inherited

3.5 Search recommendation (DBA, reaffirmed by this design)

NO NEW PAID SERVICE — COST-AVOIDANCE RECOMMENDATION Postgres full-text search (tsvector + GIN), not a dedicated search service (Algolia, Meilisearch, Elasticsearch). At 2 brands / dozens of SKUs, Postgres FTS plus the facet indexes above (category/brand/price/stock) is fully sufficient for both relevance search and faceted browse, with no new moving parts, no new operational/security surface, and no recurring cost. If the catalogue later grows to hundreds/thousands of SKUs with heavy real-time typeahead needs, that is a future revisit requiring CIO/CEO/parent decision — not decided here.

3.6 Proportionality notes (intentionally avoided by the DBA)

  • No inventory movement/audit trail (the stock_status enum matches the order-on-demand model, not a warehouse system)
  • No product_categories many-to-many (a single category_id fits the current catalog shape)
  • No nested-set/ltree for categories (plain parent_id is enough for a 2-level tree)
  • No dedicated search service (see §3.5)
  • specs as jsonb rather than a full EAV attribute system

The one place the DBA added structure ahead of strictly proven need: product_variants as its own table — justified because retrofitting it after real pricing/orders exist would be far more disruptive than building it now.

5. Open Questions / Risks

5.1 From the DBA (data layer)

  1. Hebrew slugs — a "slug" column is an English SEO convention that doesn't map cleanly onto Hebrew names. Partially addressed in §2.2 (brands/categories get real slugs now; products/variants route by id/sku until Marketing settles a convention) — final call is a Marketing/CIO decision, not settled here.
  2. specs jsonb key-naming convention needs to be agreed at the app level (e.g. fixed key sets per category), since jsonb isn't schema-enforced. Recommend Dev and DBA agree a short written convention (e.g. a per-category key list) before the catalog UI is built against it, so the frontend isn't guessing at key names.
  3. Migrating the 7 existing seeded (unpublished) Char-Broil rows from the old free-text columns (dimensions/burners/features/price_tier) to the new structured shape is its own reviewed migration step for Dev — zero live-traffic risk since nothing is published yet.
  4. Category cycle prevention is intentionally shallow (blocks only direct self-parenting) — accepted given the 2-level hierarchy.
  5. Backup/restore — the schema-agnostic backup-db.ps1 (pg_dump) will pick up the new tables automatically, but the restore path has never been tested end-to-end. DBA recommends an actual restore drill once any schema is applied, independent of this catalog design.

5.2 From this design (Solution Architect)

  1. SEO / social-link-preview tradeoff (§2.1) — client-side-only rendering means shared product links (e.g. via WhatsApp, a very common Israeli sharing channel) won't show a rich preview, since link-preview bots don't execute JS. Flagged as a genuine future decision (Option B, build-time pre-rendering), not solved by this design; revisit once Marketing has an active plan to drive traffic to individual product links.
  2. Facet-count performance at scale — client-side facet counting is fine at dozens of SKUs; if the catalogue grows substantially, the DBA's suggested materialized view (refreshed on catalog change) is the recommended next step, not a day-one requirement.
  3. Image sizing discipline — no automatic image optimization exists in the self-hosted stack; relies on twins/Marketing manually sizing images reasonably before upload (§2.5). Worth a one-line note in whatever internal "how to add a product" instructions Dev/CIO eventually write for the twins.
  4. "In stock" wording vs. order-on-demand reality (§2.4) — flagged for Marketing sign-off on exact Hebrew copy so stock badges don't imply held inventory that doesn't exist.
  5. Studio as the only content-management path — consistent with the existing approach (no custom admin panel built), but worth the CIO confirming the twins are comfortable inserting rows across five related tables (brands → categories → products → variants → images) via Studio's table editor rather than a single guided form. Not a blocker, just a usability note for whoever trains the twins on it.
