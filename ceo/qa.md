# Mangalist — QA Log

Independent verification record, owned by the QA subagent (`.claude/agents/qa.md`). Nothing CIO (or another subagent) builds is treated as "done" until it has a passing entry here — see the parent's directive (2026-07-04): best-quality software, dev/QA split, every build gated on a bug-free QA session with real test scenarios.

## How this works
- QA writes concrete test scenarios for a given build (happy path, edge cases, anything the task explicitly required) and actually executes them — not a code-read-through.
- Bugs found are logged here with repro steps and sent back to CIO to fix; QA re-verifies after the fix before signing off.
- A build is only reported to the CEO/parent as complete once its scenarios all pass here.

**Bugs found:** None in the actual shipped code path.

**One near-miss worth flagging to CIO/Dev as a landmine, not a bug in current code:** During investigation I first tested the leads insert with an explicit `Prefer: return=representation` header (to get the inserted row back, e.g. for future use of `.select()` after `.insert()`), and that **fails** with `42501 permission denied for table leads` — because Postgres requires `SELECT` privilege on a table to satisfy `RETURNING`, and by design `anon` has *no* `SELECT` grant on `leads` (insert-only, per the migration's intent). Confirmed via direct psql: `SET ROLE anon; INSERT ... RETURNING id;` → `ERROR: permission denied for table leads`, while the same `INSERT` without `RETURNING` succeeds. This does **not** affect the current code — `script.js` calls `.insert({...})` without `.select()`, and supabase-js's `insert()` does not add `return=representation` unless `.select()` is chained (confirmed by reading the actual `@supabase/supabase-js@2` bundle source), so the live contact form is unaffected. Flagging so that if anyone later adds `.select()` after `.insert()` on `leads` (e.g. to show a confirmation ID client-side), it will silently break with a permission error — by design of the insert-only RLS model, not a schema bug. Worth a code comment near the insert call in `script.js` if CIO wants to guard against a future regression.

**Scope note:** Did not test contact form through an actual browser DOM/click flow (no headless browser tooling set up in this environment) — verified the equivalent and more precise thing instead: the exact `supabase-js` client call the form's submit handler makes, executed via Node against the same live local stack, plus confirmed the HTML form's `required` attributes and script load order (`supabase-config.js` before `script.js`) by reading `index.html` directly. If CIO wants a literal click-through-the-page browser test, that would need Playwright/Puppeteer added to the toolchain first.
